# 集成测试与报告撰写 — 设计规格

## 背景

项目当前进展：第1-5步已完成（业务理解、架构设计、项目骨架、后端核心、前端核心），第6步（集成测试）和第7步（报告撰写）未开始。

核心功能代码已全部编写完成，但尚未在真实环境中验证端到端流程。需要确保"上传模板 → 创建项目 → 变量去重 → 填变量 → 生成文档 → 下载"主流程可用。

## 目标

1. 验证后端所有 API 端点功能正确
2. 验证前端端到端交互流程可用
3. 发现并修复 P0/P1 级别 Bug
4. 完成 ≤10 页项目报告

## 方案

**API 优先验证 → 前端联调 → 报告撰写**，测试与报告交叉推进。

### Phase 1：后端 API 验证

按依赖顺序逐个验证后端 API：

| 序号 | API | 方法 | 验证点 |
|------|-----|------|--------|
| 1 | `/api/v1/health` | GET | 服务可用 |
| 2 | `/api/v1/templates` | POST | 上传 DOCX，变量自动解析 |
| 3 | `/api/v1/templates` | GET | 分页、关键词搜索 |
| 4 | `/api/v1/templates/{id}` | GET | 模板详情含版本信息 |
| 5 | `/api/v1/templates/{id}/variables` | GET | 变量列表正确提取 |
| 6 | `/api/v1/projects` | POST | 创建项目，关联多模板 |
| 7 | `/api/v1/projects/{id}/deduplicated-variables` | GET | 跨模板变量去重逻辑 |
| 8 | `/api/v1/contracts/preview` | POST | 变量替换预览文本 |
| 9 | `/api/v1/contracts` | POST | 生成 DOCX 文件 |
| 10 | `/api/v1/contracts/{id}/export` | GET | 文件下载 |
| 11 | `/api/v1/contracts/batch` | POST | Excel 批量生成 |

**边界测试**：
- 上传非 DOCX 文件 → 400
- 查询不存在的 ID → 404
- 创建项目选 0 个模板
- 变量值为空时的生成行为
- 重复上传同名模板

### Phase 2：前端联调验证

浏览器中验证完整交互流程：

1. 首页统计卡片加载数据
2. 模板管理页：上传、搜索、查看变量、删除
3. 合同生成页：Steps 流程（创建项目 → 选模板 → 填变量 → 生成下载）

### Phase 3：报告撰写（与 Phase 1-2 交叉）

报告 ≤10 页，9 个章节：

| 章节 | 内容 | 页数 | 撰写时机 |
|------|------|------|---------|
| 1. 业务理解 | 签字页业务痛点、解决方案 | ~1 | 后端验证后 |
| 2. 软件架构 | 前后端架构图、技术选型理由 | ~1.5 | 后端验证后 |
| 3. 数据模型 | 核心表结构、ER 关系 | ~1 | 后端验证后 |
| 4. 关键接口/流程 | API 设计、业务流程图 | ~1.5 | 后端验证后 |
| 5. 实现权衡 | MVP 简化内容、理由、扩展方向 | ~1.5 | 全部验证后 |
| 6. 测试/验证结果 | 测试过程、通过用例、Bug 修复 | ~1.5 | 前端联调后 |
| 7. 未解决问题 | 已知限制 | ~0.5 | 全部验证后 |
| 8. 如果再给一个月 | 可扩展功能 | ~0.5 | 全部验证后 |
| 9. AI 工具使用记录 | 各阶段 AI 使用、错误、验证 | ~1 | 最后撰写 |

### Bug 修复策略

| 级别 | 定义 | 处理方式 |
|------|------|---------|
| P0 | 阻塞核心流程（API 500、文件生成异常） | 立即修复 |
| P1 | 功能缺陷（数据错误、逻辑遗漏） | 本轮修复 |
| P2 | 体验问题（UI 样式、提示文案） | 仅记录，不修复 |

所有修复记录到 CLAUDE.md Progress Log。
